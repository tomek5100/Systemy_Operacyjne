Zadanie 1. Biblioteka dynamiczna libshmem.so

Napisać bibliotekę dynamiczną (dzieloną) libshmem.so zawierającą implementacje następujących funkcji obsługujących pamięć współdzieloną standardu POSIX:

  • int shmCreate(const char* name, int size);
    Funkcja tworzy pamięć dzieloną o nazwie name i rozmiarze size.
    Funkcja zwraca deskryptor obiektu pamięci dzielonej.
  • int shmOpen(const char* name);
    Funkcja otwiera obiekt pamięci dzielonej i zwraca jego deskryptor.
  • void shmRm(const char* name);
    Usuwa obiekt pamięci dzielonej o nazwie name.
  • void* shmMap(int fd, int size);
    Odwzorowuje obiekt pamięci dzielonej o deskryptorze fd i rozmiarze size w przestrzeń adresową procesu.
    Funkcja zwraca adres początku odwzorowanego obszaru pamięci dzielonej.
  • void shmClose(void *ptr, int fd, int size);
    Usuwa odwzorowanie obiektu pamięci dzielonej i zamyka powiązany z nim deskryptor.
  • int shmSize(int fd);
    Zwraca rozmiar pamięci dzielonej określonej deskryptorem fd.
    Proszę użyć funkcji systemowej fstat.
  • void shmInfo(int fd);
    Wypisuje informacje o pamięci dzielonej: prawa dostępu, rozmiar, właściciel, etc.

Funkcje mają korzystać z odpowiednich funkcji systemowych i obsługiwać wszystkie możliwe błędy przez nie zwracane (wypisać odpowiedni komunikat (perror) i zakończyć działanie (exit)).
Należy napisać odpowiedni plik nagłówkowy shmem.h!
Aby utworzyć bibliotekę dzieloną libshmem.so należy:

  1. Skompilować plik źródłowy shmem.c do pliku obiektowego shmem.o z opcją -fPIC (Position-Independent Code):
gcc -c -O -fPIC shmem.c
  1. Utworzyć bibliotekę dzieloną libshmem.so:
gcc -shared -o libshmem.so shmem.o
  1. Aby użyć własnej biblioteki libshmem.so należy dodać ją opcją -l :
gcc shmemuse.c -L. -lshmem

Powyższe operacje mają być wykonywane automatycznie w pliku Makefile.

Aby uruchomić program używający biblioteki dzielonej należy dodać do zmiennej środowiskowej LD_LIBRARY_PATH ścieżkę do katalogu zawierającego bibliotekę:

export LD_LIBRARY_PATH=.

Uwaga: Każda komenda w pliku Makefile wywołuje osobną powłokę (zmienna SHELL), dlatego polecenie export nie zadziała i trzeba ją ręcznie wywołać w powłoce.

Program przyjmuje jednoliterowy argument i ewentualnie drugi argument:

  • ./shmuse.x c size - Tworzy segment pamięci dzielonej o rozmiarze size bajtów.
  • ./shmuse.x d - Usuwa segment pamięci dzielonej

Zadanie 3. Producent i konsument

Przy pomocy pamięci dzielonej oraz semaforów standardu PSOIX zaimplementować problem ,,Producenta i Konsumenta’’ dla N-elementowego bufora cyklicznego (tzn. po dojściu do końca bufora wracamy na jego początek) umieszczonego w pamięci dzielonej, gdzie elementem bufora jest pewna ustalona porcja bajtów. Dla zademonstrowania, że nie doszło do utraty ani zwielokrotnienia towaru, niech Producent pobiera ,,surowiec’’ (np. porcje bajtów lub liczby typu int) z pliku tekstowego magazyn.txt i wstawia go jako towar do bufora, a Konsument niech umieszcza pobrany z bufora towar w innym pliku tekstowym schowek.txt. Po zakończeniu działania programów (wyczerpaniu zasobów ,,surowca") oba pliki tekstowe powinny być identyczne. Oba procesy niech drukują odpowiednie komunikaty na ekranie, w tym towar, który przesyłają. Do zasymulowania różnych szybkości działania programów użyć funkcji usleep np. z losowym czasem usypiania. Należy użyć dwóch semaforów, pierwszy blokuje zapis (producent) gdy bufor cykliczny jest pełny, drugi blokuje odczyt (konsument) gdy bufor cykliczny jest pusty. Struktura bufora cyklicznego, która znajduje się w pamięci dzielonej może mieć postać:

struct cykliczny { 
   int size;            // Całkowity rozmiar bufora 
   int beg;             // Pierwszy element do odczytu 
   int end;             // Element za ostatnim zapisanym elementem 
   towar buf[ROZMIAR];  // Dane, własny typ 'towar' może być równoważny np. 'int' 
};

Należy użyć własnych bibliotek libsem.a i libshmem.so. Implementacje producenta (producent.c) i konsumenta (konsument.c) umieścić w oddzielnych programach, a uruchamiać je przez funkcje fork() i exec() z dodatkowego procesu macierzystego (glowny.c), który na początku powinien utworzyć i zainicjować semafory oraz pamięć dzieloną, a po zakończeniu procesów potomnych je usunąć. Wydzielić funkcje, które zapisują i odczytują z wspólnego bufora cyklicznego.

Wskazówki

  • Do synchronizacji jednego producenta i jednego konsumenta wystarczą dwa semafory ogólne: jeden do wstrzymywania producenta kiedy bufor jest pełny (inicjowany liczbą elementów bufora), a drugi do wstrzymywania konsumenta kiedy bufor jest pusty (inicjowany zerem).
  • Synchronizacja powinna dotyczyć dostępu do pojedynczego elementu bufora, a nie do całego bufora, tzn. producent i konsument mogą w tym samym czasie operować na różnych elementach bufora.
  • Dla wielu producentów i konsumentów wystarczą dwa dodatkowe semafory binarne - do wzajemnego wykluczania każdego typu procesów.

Pamięć dzielona standardu POSIX - wskazówki

Pamięć dzielona jest najszybszą metodą komunikacji pomiędzy niezależnymi procesami, zwłaszcza w przypadku przekazywania dużej ilości danych.

Oryginalny X Window System, czyli system okien (graficzny interfejs użytkownika) pod Linuxem, wykorzystywał mechanizm TCP do komunikacji pomiędzy klientem (aplikacjami) a serwerem wyświetlania. Takie rozwiązanie jest oczywiście powolne, szczególnie w przypadku pracy na jednym komputerze (czyli zazwyczaj, choć X Window pozwala na transparentną pracę zdalną np. przez SSH).

Nowoczesny serwer wyświetlania Wayland wykorzystuje do komunikacji z klientami model pamięć współdzielonej standardu POSIX (pliki mapowane w pamięci, ang. file-backed shared memory model). Mapowany plik nie musi być zapisany na dysku - co więcej - nie powinien i nie jest, ale deskryptor do otwartego pliku jest przekazywany przez klienta do serwera. Plik jest jedynie plikiem tymczasowym utworzonym, np. w wyniku wywołania funkcji mkstemp, i jest usuwany jeszcze w trakcie korzystania z pamięci dzielonej, aby inne procesy (nawet tego samego użytkownika) nie miały do niego dostępu.

Ciekawostka: Utworzona pamięć dzielona standardu POSIX jest widoczna jako plik o podanej nazwie i odpowiednich prawach dostępu w katalogu /dev/shm (pseudo system plików tmpfs).

Ciekawostka:
Parametr ochrony (ang. protection option) PROT_NONE jest wykorzystywany między innymi do ochrony przed atakami typu buffer overflow (przepełnienie bufora) poprzez wprowadzenie mechanizmu guard pages. Jest wykorzystywany przez wirtualne maszyny, interfejsy sieciowe, interpretery, …

Schemat programu, który tworzy pamięć współdzieloną standardu POSIX i do niej pisze

Uwaga: Należy obsłużyć możliwe błędy dla każdego wywołania funkcji systemowych.

const char *name = "/shared-test";
const int size = 1024;

// Tworzy pamięć dzieloną widoczną jako plik /dev/shm/shared-test

int fd  = shm_open(name, O_RDWR | O_CREAT | O_EXCL, 0666);

// Zmiana rozmiaru pamięci dzielonej / pliku

    ftruncate(fd, size);

// Odwzorowanie pamięci dzielonej w przestrzeni adresowej procesu

char *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// Zapis do pamięci dzielonej pod adresem ptr

    strcpy(ptr, "Hello World in file-backed shared memory");

// Usunięcie odwzorowania

    munmap(ptr, size);

// Zamknięcie deskryptora

    close(fd);

Schemat programu, który czyta z pamięci współdzielonej i ją usuwa

Uwaga: Należy obsłużyć możliwe błędy dla każdego wywołania funkcji systemowych.

const char *name = "/shared-test";
const int size = 1024;

// Otwarcie obiektu pamięci dzielonej tylko do odczytu

int fd  = shm_open(name, O_RDONLY, 0666);

// Odwzorowanie pamięci dzielonej w przestrzeni adresowej procesu

char *ptr = mmap(NULL, size, PROT_READ, MAP_SHARED, fd, 0);

// Wypisanie zawartości pamięci dzielone (odczyt z pamięci)

    printf("Shared data: %s\n", ptr);

// Usunięcie odwzorowania

    munmap(ptr, size);

// Zamknięcie deskryptora

    close(fd);

// Usunięcie obiektu pamięci dzielonej

    shm_unlink(name);
